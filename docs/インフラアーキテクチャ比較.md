# インフラアーキテクチャ比較設計書

> **作成日:** 2026-03-04
> **目的:** Phase 1 着手前のインフラ技術選定の最終確認。Google + Supabase を基準軸に、OTP2 デプロイを主要変数として各アーキテクチャを比較する。

---

## 1. 前提・設計上のポイント

### 1.1 コンポーネント構成

本プロジェクトで必要なインフラコンポーネントは以下の通り。

| コンポーネント | 役割 |
|---|---|
| **FastAPI バックエンド** | REST API サーバー（認証・スケジュール・提案等） |
| **OTP2 サーバー** | 経路探索エンジン（Java, GraphQL API）|
| **PostgreSQL DB** | ユーザー・スケジュールデータ永続化 |
| **フロントエンド CDN** | SPA ホスティング |
| **FCM** | プッシュ通知 |

### 1.2 OTP2 が選定の最大変数

FastAPI・DB・フロントエンドはいずれも無料枠内に収まるが、**OTP2 のデプロイ先が月次コストの大半を占める**。

OTP2 の特性上、デプロイ先には以下の制約がある。

| 要件 | 理由 |
|---|---|
| **常時起動（スリープ不可）** | graph.obj (709 MB) をメモリに展開しており、コールドスタートに2〜5分かかる |
| **RAM 2〜4 GB 必須** | graph.obj 709 MB + JVM ヒープ + ルーティング演算バッファ |
| **ステートフル（永続化）** | graph.obj のディスク保持が必要 |
| **GraphQL on 8080** | FastAPI が `OTP2_GRAPHQL_URL` 環境変数で参照 |

このため、Cloud Run（ゼロスケール型）や Railway（コスト高）は OTP2 に適さない。

## 2. OTP2 デプロイ選択肢の個別比較

### 2.1 GCE e2-medium（推奨 A）

| 項目 | 内容 |
|---|---|
| スペック | 2 vCPU（共有）/ 4 GB RAM |
| 常時起動 | ✅（VM は停止しない限り継続稼働） |
| RAM 十分性 | ✅（4 GB は OTP2 に十分。余裕あり） |
| ストレージ | 20 GB SSD ディスク $4.42/月 |
| 月額コスト | **$31.38 + $4.42 ≒ $36/月**（Tokyo: asia-northeast1） |
| セットアップ難度 | 中（SSH ログイン → systemd でサービス化） |
| OTP2起動方法 | systemd unit で `java -jar otp.jar --serve data/` を自動起動 |
| リージョン | asia-northeast1（東京） |

**メリット**
- RAM 余裕あり → OOM リスクなし
- GCP 統合（Cloud Logging・Cloud Monitoring で FastAPI と同じ画面で監視可）
- VM 再起動ポリシーで crash 時の自動復旧設定が容易
- GCP のマネージド環境で SLA が比較的高い

**デメリット**
- 3 サービス（Cloud Run + GCE + Supabase）の管理が必要
- GCE の SSH セットアップ・ファイアウォール設定が必要
- FastAPI と OTP2 が別 VPC 扱いになるため VPC 内通信不可（外部 IP で通信）

---

### 2.2 fly.io shared-cpu-2x 4GB（推奨 B）

| 項目 | 内容 |
|---|---|
| スペック | 2 vCPU（共有）/ 4 GB RAM |
| 常時起動 | ✅（Machine を起動したままにする設定が可） |
| RAM 十分性 | ✅（4 GB で余裕あり） |
| ストレージ | persistent volume 2 GB $0.30/月 |
| 月額コスト | **$22.22（iad 基準）+ 東京 nrt プレミアム約 30% ≒ $29/月** + volume $0.30/月 ≒ **$30/月** |
| セットアップ難度 | 低〜中（`flyctl` + Dockerfile + `fly.toml` のみ） |
| OTP2 起動方法 | Dockerfile で OTP2 JAR を COPY → volume マウントで graph.obj 参照 |
| リージョン | nrt（東京） |

**メリット**
- `flyctl deploy` 1コマンドでコンテナデプロイ完了
- Dockerfile ベースで CI/CD 対応しやすい
- 東京リージョン（nrt）対応でレイテンシ低い
- GCE より VM 管理が少ない（SSH 不要、コンテナ abstraction）
- persistent volume でグラフファイルを保持（Docker イメージを軽量に保てる）

**デメリット**
- fly.io と GCP の 2 サービス管理が必要
- shared CPU なので OTP2 の経路計算が集中すると GCE より遅い可能性がある
- fly.io のサポートは GCP より手薄
- 東京プレミアム込みのコストは GCE と大差なくなる（$29 vs $31）

**OTP2 graph.obj の扱い（fly.io 版）**

```
Option 1: Docker image に graph.obj を COPY で焼き込む
  - メリット: シンプル、volume 不要
  - デメリット: イメージサイズ ~1 GB（プッシュに時間がかかる）
  - GTFS 更新時は全リビルド必要

Option 2: persistent volume に graph.obj を置く（推奨）
  - イメージは OTP2 JAR (~200 MB) のみ
  - 初回のみ volume に graph.obj をコピー（flyctl sftp / ssh コマンド）
  - GTFS 更新時はボリュームの graph.obj を差し替えるだけ
```

| 項目 | 内容 |
|---|---|
| スペック | 1 vCPU（共有）/ 2 GB RAM |

**メリット**

**デメリット**

### 2.4 GCE Spot VM e2-medium（格安・不安定）

| 項目 | 内容 |
|---|---|
| 月額コスト | e2-medium Tokyo 通常 $31.38 の約 56% OFF ≒ **$14/月** |
| リスク | GCP に随時終了される可能性あり（30秒通知のみ） |
| OTP2 影響 | 終了後の再起動で graph 読み込み（1〜2 分の停止）が発生 |

**適用場面**: デモ頻度が低い開発期間中。デモ当日・直前は通常 VM に切り替え推奨。

---

### 2.5 GCP Cloud Run（OTP2 用途では非推奨）

| 項目 | 内容 |
|---|---|
| min-instances=1 + 1 vCPU + 2 GB RAM | ~$57/月（always-on CPU: $0.000018/vCPU-秒） |
| min-instances=1 + 2 vCPU + 4 GB RAM | ~$104/月 |
| コールドスタート | 2〜5 分（graph.obj 709 MB の JVM 読み込み） |

**結論: Cloud Run は OTP2 に不適。** ゼロスケールと相性が悪く、min=1 を設定すると GCE より高額になる。

---

### 2.6 Railway Hobby（OTP2 用途では非推奨）

| 項目 | 内容 |
|---|---|
| OTP2 目安（1 vCPU + 2 GB, 常時起動） | CPU $20.02 + RAM $20.02 - $5 クレジット = **約 $40/月** |
| OTP2 目安（1 vCPU + 4 GB, 常時起動） | CPU $20 + RAM $40 - $5 クレジット = **約 $55/月** |

**結論: Railway は常時起動のメモリ集約型ワークロードに不向き。** 料金体系が割高になる。

**向いている場面**
- GCP を一元管理したい
- Cloud Logging・Cloud Monitoring で FastAPI と OTP2 をまとめて監視したい

**非機能要件対応**

| 観点 | 評価 |
|---|---|
| 可用性 | GCE は GCP SLA 99.5%。Cloud Run は 99.95%。Supabase 無料は SLA なし |

| セキュリティ | GCE はファイアウォールで 8080 を Cloud Run からのみ許可。Supabase は TLS 接続のみ |
| 監視・ロギング | Cloud Logging + Cloud Monitoring で統合管理可 |
| バックアップ | Supabase 無料は日次バックアップなし（手動エクスポート必要） |
| コスト予測精度 | GCE は固定料金で予測しやすい |

### 4.5 外部 API コスト試算

#### WeatherAPI.com
- 無料枠: **1,000,000 API コール/月**
- ハッカソン試算: ユーザー数 100 × 1 日 5 回 × 30 日 = 15,000 コール/月
- **コスト: $0**（無料枠の 1.5%）

#### Gemini API（AI 提案）
- 無料枠: Gemini 2.0 Flash = 15 RPM / 1,500 RPD（1日1,500リクエストまで）
- 試算: 1 リクエストあたり ~1,000 input tokens + ~500 output tokens
- ハッカソン試算: 100 ユーザー × 2 回/日 × 30 日 = 6,000 リクエスト/月
- 無料枠（1,500 RPD = 45,000/月）で余裕でカバー
- **コスト: $0**（無料枠内）
- 万一有料になった場合: 6,000 リクエスト × (1,000 input × $0.10/1M + 500 output × $0.40/1M) ≒ **$1.80/月**

#### Supabase（万一 Pro Plan が必要になった場合）
- 現在の無料枠: 500 MB DB ストレージ・7 日間無活動でポーズ
- Pro Plan: **$25/月**（ポーズなし・8 GB ストレージ）
- ハッカソン期間中は無料枠で十分（スキーマ + 少量データ）
- **本番運用継続なら Pro Plan への移行を検討**

## 6. Supabase 無料枠の注意点

ハッカソン開発中に Supabase 無料プロジェクトが **7 日間無活動でポーズ** される点に注意が必要。

| タイミング | 対策 |
|---|---|
| 開発期間中 | 週 1 回以上 API リクエストを飛ばすか、定期的にダッシュボードにアクセスして手動再開する |
| ハッカソン当日 | 前日に必ずダッシュボードでアクティブ状態を確認する |
| 長期運用する場合 | Supabase Pro ($25/月) へアップグレードするか、Railway PostgreSQL / Neon に移行する |

## 7. 推奨・結論

### 推奨: **Plan B（fly.io 4GB OTP2 + GCP Cloud Run + Supabase）**

| 理由 | 詳細 |
|---|---|
| **コストと安定性のバランス最良** | Plan A より $7/月安く、OTP2 の 4GB RAM で安定性も維持 |
| **セットアップが早い** | `flyctl deploy` でコンテナデプロイ完了。SSH 不要 |
| **Dockerfile でバージョン管理** | OTP2 の設定・起動コマンドをコードで管理できる |
| **GTFS 更新が容易** | persistent volume の graph.obj を差し替えるだけで GTFS 更新できる |
| **東京リージョン対応** | fly.io `nrt` で国内ユーザーへのレイテンシを最小化 |

**ただし GCP に慣れているチームなら Plan A でも問題なし**（$7/月の差で管理を統一できる利点がある）。

### 代替推奨: **Plan A（GCE e2-medium + GCP Cloud Run + Supabase）**

GCP のコンソール・監視ツールに慣れている場合は Plan A が管理面でシンプル。

### 非推奨
- Cloud Run での OTP2 ホスト（コールドスタート問題・コスト過大）
- Railway での OTP2 ホスト（RAM コスト過大）
- fly.io 2GB OTP2（OOM リスク）
