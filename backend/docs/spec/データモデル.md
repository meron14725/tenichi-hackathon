### ScheduleList（予定リスト）

「1日の予定をまとめた名前付きリスト」。カレンダー上に表示され、所属する Schedule 群と持ち物リスト（PackingItem）を持つ。Template が「再利用可能な雛形」であるのに対し、ScheduleList は「実際の日付に紐づく予定のグルーピング」。

| カラム | 型 | 備考 |
|--------|----|----|
| id | BIGINT PK | |
| user_id | BIGINT FK → User | |
| name | VARCHAR(100) | 予定リスト名（例: "出社の日", "デートの日"） |
| date | DATE | 対象日付 |
| created_at | TIMESTAMP | |
| updated_at | TIMESTAMP | |

**対応エンドポイント:** `GET/POST /schedule-lists`, `GET/PUT/DELETE /schedule-lists/{id}`

---

| カラム | 型 | 備考 |

### ScheduleRoute（選択ルート）

予定作成時にモバイルアプリが OTP2 経路探索 API を叩き、返された複数ルート候補からユーザーが選択した1つのルートを保持する。Schedule と 1:1。

| カラム | 型 | 備考 |
|--------|----|----|
| id | BIGINT PK | |
| schedule_id | BIGINT FK → Schedule | UNIQUE。1予定につき1ルート |
| route_data | JSONB | OTP2 から返された itinerary の JSON 全体を保存 |
| departure_time | TIMESTAMP | 出発時刻（検索用に非正規化） |
| arrival_time | TIMESTAMP | 到着時刻（検索用に非正規化） |
| duration_minutes | INT | 総所要時間（分）（検索用に非正規化） |
| created_at | TIMESTAMP | |

> **設計メモ:** `route_data` に itinerary JSON 全体を JSONB で保存することで、legs（区間情報）を含むルート詳細をそのまま表示に使える。`departure_time` / `arrival_time` / `duration_minutes` は一覧表示時のソート・フィルタ用に非正規化して保持。

**対応エンドポイント:** `POST /schedules/{id}/route`, `GET /schedules/{id}/route`, `DELETE /schedules/{id}/route`

## 確定した設計判断

| 項目 | 判断 | 理由 |
|------|------|------|
| `template_id` を Schedule に持つか | **持たない** | テンプレ適用はコピーのみ。テンプレ編集が既存予定に波及しない |
| タグ管理 | **Tag + 中間テーブル（ScheduleTag / TemplateScheduleTag）** | Schedule と TemplateSchedule を独立して多対多管理 |
| DeviceToken | **別テーブル** | 1ユーザー複数端末を想定 |
| UserSettings | **User と別テーブル（1:1）** | プロフィールと設定を分離し、取得・更新を独立して行えるようにする |
| 天気データ | **DBに保存しない** | BFF として外部API（WeatherAPI.com 等）をリクエスト時に都度呼び出す |
| 選択ルートの保持 | **ScheduleRoute テーブルに JSONB で保存** | モバイルアプリが OTP2 で取得した複数ルート候補からユーザーが選択した1つを保持。itinerary JSON 全体を `route_data` に保存し、主要フィールドは検索用に非正規化 |
| 認証方式 | **JWT のみ** | アクセストークン + リフレッシュトークン。シンプル、ハッカソン向き |
| UserSettings / NotificationSettings 初期化 | **register 時一括作成** | デフォルト値をそれぞれ設計し、同一トランザクションで生成。null チェック不要でシンプル |
| タグのスコープ | **グローバル（全ユーザー共通）** | `Tag.name UNIQUE` を維持 |
| タグ作成権限 | **事前定義のみ（seed データ投入）** | `POST /tags` は実装しない |
| `Schedule.travel_mode` の nullable | **nullable** | 目的地なし予定（自宅リモート等）に対応 |
| 終日フラグ（`is_all_day`）| **追加しない** | ハッカソンスコープ外 |
| ソフトデリート | **物理削除** | `deleted_at` カラム追加なし。実装をシンプルに保つ |
| パスワードリセット | **スコープ外** | `PasswordResetToken` テーブル不要 |
| 繰り返し予定 | **`recurrence_rule VARCHAR(500)` を nullable で追加** | 現時点は非対応。将来 RRULE 形式で使用できるよう拡張可能な設計にする |
| リマインダー設定 | **1つのみ（`reminder_minutes_before`）** | 現在は INT 1つ。将来の複数設定拡張を意識したコード設計にする |
| `reminder_minutes_before` の配置 | **`UserSettings` に配置（登録時必須入力）** | 登録時に必須入力。`NotificationSettings` から移動 |
| Template のコンセプト | **1日全体の予定集合の雛形（再利用用）** | Template 自体は名前と TemplateCategory のみ保持。複数の TemplateSchedule（予定の雛形）を内包 |
| 予定リスト（ScheduleList）のコンセプト | **実際の日付に紐づく予定のグルーピング** | Template（再利用可能な雛形）とは別概念で共存。1日の予定に名前をつけてカレンダーに表示する。持ち物リスト（PackingItem）を内包 |
| Template と ScheduleList の関係 | **共存（別概念）** | Template = 再利用可能な雛形、ScheduleList = 実際の日の予定グループ。Template apply 時に ScheduleList を自動生成することも可能 |
| TemplateSchedule の時刻 | **固定時刻（HH:MM 形式）** | 日付なし TIME 型。apply 時に指定日付と組み合わせて TIMESTAMP の Schedule を生成 |
| Template のタグ管理 | **TemplateScheduleTag（TemplateSchedule と Tag の中間テーブル）** | Template 直接へのタグ付けは廃止（旧 TemplateTag を削除） |
| Template の種別分類 | **TemplateCategory マスタ** | Tag ではなく専用の TemplateCategory で分類。seed 値: **"仕事の日" / "在宅勤務" / "休日"（確定）** |
| 経路 API | **OTP2（検証中）** | Phase 7 で詳細設計。現在選定中のためブロッカー状態 |
| Suggestions（提案ロジック）| **Gemini API を使って実装（確定）** | `GET /suggestions/today`（今日の提案）と `GET /suggestions/{schedule_id}`（予定ごとの提案）の 2 エンドポイント |
